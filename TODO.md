## Сшивка геодаты на границах регионов

**Проблема.** `generate_region_full` читает только свой `Maps/XX_YY.unr` и
интерполирует рельеф внутри своего heightmap. На стыке регионов это даёт два
класса дефектов, видимых в игре ровно по линиям квадратов:

1. **Акторы через границу.** Меш (мост, скала, здание), стоящий в соседнем
   `.unr`, но физически нависающий над этим регионом, не попадает в геодату —
   дыра в проходимости посреди объекта. Канонический случай — мост между
   регионами: половина коллизии теряется.
2. **Шов рельефа.** `terrain_cells` клампит крайние тексели heightmap
   (`geolib/generate.py`, цикл по `CELLS`) — краевые ячейки получают высоту
   «плоского продолжения» вместо реального склона в соседний регион. Обе стороны
   шва считаются независимо → ступенька по линии границы; при перепаде > `UP_STEP`
   (16) — невидимая стена, полоса застревания мобов вдоль кромки квадрата.

**Что сделать.**
- Меши: при сборке региона подгружать `StaticMeshActor` из четырёх соседних
  `.unr` и включать их треугольники, если bbox актора пересекает полосу шириной
  ~макс. радиус меша от границы. Фильтр по bbox, чтобы не тащить весь соседний
  регион.
- Рельеф: при интерполяции краевых ячеек читать одну краевую строку/столбец
  heightmap соседних четырёх регионов вместо клампа собственного тексель.
- Учесть classic-зоны (соседи могут быть `XX_YY_Classic.unr`).

**Как проверить.** `geotool diff <сборка_с_соседями> <сборка_без>` — все
изменившиеся ячейки должны лечь полосой вдоль границ региона; так же видно точное
их количество. Финальная проверка — пройти мосты/склоны на стыках квадратов в игре
(эталонной геодаты не существует, ground truth — только коллизия клиента).

## Точность стен в углу ячейки (некритично, вероятно не делать)

**Проблема.** Проверка стены (`_wall_between`, `geolib/generate.py`; дублируется
в `raycast_ti.py` `seg_hit`/`wall_any`) сэмплирует границу ячейки тремя линиями
(`WALL_LAT_OFFS = -7/0/+7`), оставляя по ~1 юниту у каждого края слепыми. Тонкий
объект (фонарь, столб, ствол), стоящий ровно в углу, может не задеть ни одну
линию → сторона остаётся открытой.

**Почему, скорее всего, не стоит трогать.**
- Промах бьёт только по тонким объектам; широкие стены (здания, заборы) из
  крупных треугольников ловятся центральной/±7 линиями всегда.
- Пройти сквозь фонарь — косметика, никто об него не застревает и путь не строит.
- Anti-глухая логика (`generate.py`, блок «тонкий объект не запирает клетку»)
  и так намеренно НЕ даёт столбам/стволам перегораживать ячейку — «ловить их
  точнее» частично противоречит этому замыслу.

**Если всё же понадобится** (в игре нашлись реальные просачивания сквозь угловые
стены):
- Дёшево: загустить `WALL_LAT_OFFS` до `-7.5/-3.75/0/3.75/7.5` — одна строка,
  оба пути (Python и Taichi) читают эту же константу, паритет CPU/GPU сохраняется
  сам. Слепые края → ±0.5 юнита.
- Точно: заменить сэмплирование точками на тест пересечения стенового
  треугольника с прямоугольником границы (16 × окно корпуса 10..32). Убирает
  «почти» совсем, но это переписывание обеих синхронных реализаций + ре-валидация
  байт-паритета.
